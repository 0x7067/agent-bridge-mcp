## MODIFIED Requirements

### Requirement: Provider adapters preserve public provider capabilities
The system SHALL expose the same public provider names, supported modes, and provider-specific option values through the provider registry that are currently exposed through `providers_list` and agent validation.

#### Scenario: Listing providers
- **WHEN** a caller invokes `providers_list`
- **THEN** the response includes `claude`, `cursor`, `kimi`, and `codex` with their supported modes and provider-specific options unchanged.

#### Scenario: Validating provider-specific options
- **WHEN** a caller previews or spawns an agent with provider-specific fields such as `effort` or `thinking`
- **THEN** validation accepts only the values supported by the selected provider and rejects unsupported values before command construction.

### Requirement: Provider adapters build stable task commands
The system SHALL build provider command descriptors through provider adapters while preserving public agent behavior, cwd, timeout, prompt rendering semantics, agent metadata behavior, and preview redaction. Provider adapters MAY change internal command arguments, prompt transport mechanisms, or launch strategy metadata when required to improve provider safety or reliability, provided the public MCP agent lifecycle behavior remains compatible.

#### Scenario: Previewing a task command
- **WHEN** a caller invokes `agent_preview` for any supported provider and mode
- **THEN** the command descriptor is generated by the selected provider adapter and the preview response preserves current prompt redaction and environment key behavior.

#### Scenario: Spawning a task command
- **WHEN** a caller invokes `agent_spawn` for any supported provider and mode
- **THEN** the task manager launches the command descriptor generated by the selected provider adapter and preserves the public agent lifecycle response behavior.

#### Scenario: Hardening internal provider transport
- **WHEN** a provider adapter changes internal command arguments or prompt transport to avoid data leakage or improve reliability
- **THEN** the adapter preserves public prompt semantics, preview redaction, cwd, timeout, agent metadata, and lifecycle response compatibility.

#### Scenario: Using adapter-owned launch strategy
- **WHEN** a provider adapter marks a command for a provider-specific launch strategy such as Claude host-runner execution
- **THEN** preview, smoke checks, and agent spawning use that launch strategy without changing the public agent lifecycle API shape.

### Requirement: Provider adapter refactor is public-API preserving
The system SHALL keep the canonical public MCP tool surface and agent lifecycle response shapes unchanged during provider adapter refactors.

#### Scenario: Listing MCP tools
- **WHEN** a caller invokes `tools/list`
- **THEN** the same canonical `agent_*` tool names and accepted public argument names remain available.

#### Scenario: Reading task lifecycle results
- **WHEN** a caller uses agent lifecycle tools such as `agent_status`, `agent_logs`, `agent_wait`, or `agent_result`
- **THEN** their response shapes remain compatible with the canonical simplified agent API.
